================================================================================
  UrokPlus — авторизация и разграничение по ролям (кратко для экзамена)
================================================================================

1. ОБЩАЯ ИДЕЯ
-------------
  Реализована классическая схема «логин + пароль» с хранением учётных записей
  в PostgreSQL. После успешного входа клиент сохраняет идентификатор пользователя
  и дальше передаёт его на сервер в HTTP-заголовке. Отдельного JWT/OAuth в
  проекте нет: доверие к запросу строится на том, что клиент присылает свой
  числовой id в заголовке X-User-Id (для учебного проекта; в продакшене обычно
  добавляют подписанный токен).


2. БАЗА ДАННЫХ (PostgreSQL)
----------------------------
  Таблица users (см. database/init.sql):
    - login — уникальный логин;
    - password_hash — хэш пароля (на сервере используется SHA-256);
    - role — роль: одна из строк STUDENT, PARENT, TEACHER, ADMIN;
    - ограничение CHECK допускает только эти четыре значения.

  Роли закреплены и на уровне схемы БД, и в коде клиента (enum UserRole).


3. СЕРВЕР (Node.js + Express, server/index.js)
---------------------------------------------

  3.1. Вход
    POST /api/auth/login
    - Тело: JSON { login, password }.
    - Пароль на сервере хэшируется (sha256), выполняется запрос:
      SELECT id, login, role FROM users
      WHERE LOWER(login) = LOWER($1) AND password_hash = $2
    - При успехе: JSON { success: true, user: { id, login, role } }.
    - При неверных данных: 401 и сообщение об ошибке.

  3.2. Регистрация через приложение
    Публичная регистрация отключена (403). Создание пользователей админом —
    отдельный эндпоинт POST /api/admin/register (см. ниже).

  3.3. Идентификация последующих запросов
    Большинство методов читает пользователя из:
      req.headers['x-user-id'] или query userId
    То есть «кто я» = числовой id, переданный клиентом.

  3.4. Разграничение по ролям на сервере (примеры)
    - Админ: POST /api/admin/register
        Сначала по X-User-Id загружается role из users.
        Если role !== 'ADMIN' → 403 «Требуется роль ADMIN».
        Иначе создаётся новый пользователь с ролью из тела запроса
        (роль проверяется по белому списку: STUDENT, PARENT, TEACHER, ADMIN).

    - Родитель: GET /api/parent/children
        Только если role === 'PARENT', иначе 403 «Только для родителя».

    - Ученик: PUT /api/assignments/:id/status
        Смена статуса задания разрешена только при role === 'STUDENT',
        иначе 403.

    - Оценки: GET /api/grades
        Логика выборки зависит от роли:
          STUDENT — только свои оценки (по ФИО из profile);
          PARENT — оценки детей по таблице parent_student; при запросе за
                    конкретным ребёнком проверяется связь родитель–ученик;
          TEACHER/ADMIN (и запросы без userId в особых случаях) —
                    расширенный или полный доступ к списку оценок.

    - Задания: GET /api/assignments
        Для STUDENT в выборку подмешивается персональный статус (JOIN к
        assignment_student_status); для остальных ролей статус не возвращается.

    - Чаты и сообщения
        Доступ к чату проверяется по формату chat_id и участию userId
        (функции userParticipatesInChat, peerUserIdFromChat), а не по роли
        напрямую — чтобы пользователь не читал чужие переписки.


4. КЛИЕНТ (Android, Kotlin + Jetpack Compose)
---------------------------------------------

  4.1. Вход
    AuthRepository.login() вызывает Retrofit-интерфейс UrokPlusApi:
      POST api/auth/login с телом LoginRequest(login, password).
    При success == true и непустом user:
      - в SharedPreferences (urok_plus_prefs) сохраняются user_id и login;
      - роль из ответа преобразуется в enum UserRole (UserRole.fromString).

  4.2. Передача «сессии» в API
    Для защищённых методов в UrokPlusApi в каждый запрос явно передаётся
    заголовок @Header("X-User-Id") userId: Long — значение берётся из
    getUserId() после входа. Отдельного OkHttp-интерцептора для подстановки
    заголовка в проекте нет: id передаётся параметром в каждом вызове API.

  4.3. Интерфейс по ролям (UI)
    После успешного входа MainActivity переключает экран на MainScreen и
    передаёт userRole: UserRole.
    В MainScreen список вкладок (tabs) формируется через when(userRole):
      STUDENT, PARENT, TEACHER, ADMIN — разный набор разделов
      («Главная», «Чаты», «Журнал», «Админ» и т.д.).
    Таким образом разграничение на клиенте — прежде всего отображение разного
    функционала; критичные проверки для безопасности данных продублированы
    на сервере (см. п. 3.4).


5. ЧТО СКАЗАТЬ НА ЗАЩИТЕ (1–2 фразы)
------------------------------------
  «Авторизация — по логину и паролю, пароль хранится в БД в виде SHA-256.
  После входа клиент сохраняет id пользователя и передаёт его в заголовке
  X-User-Id. Роль хранится в PostgreSQL и проверяется на сервере для
  админских и родительских операций; в приложении по роли строится разный
  набор экранов.»


6. ОГРАНИЧЕНИЯ УЧЕБНОГО ВАРИАНТА (по желанию упомянуть)
-------------------------------------------------------
  Нет подписанного токена: зная id другого пользователя, теоретически можно
  подставить чужой X-User-Id. В реальном приложении добавили бы JWT или
  сессионные cookie и проверку прав на каждом эндпоинте единообразно.
